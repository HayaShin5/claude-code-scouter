## Context

Claude Codeはツール実行前に `PreToolUse` フックを発火する。このフックを利用して、Bashコマンドの危険度を判定し、VSCodeのステータスバーに表示するVSCode拡張を構築する。

現状、プロジェクトは新規。既存のコードベースやアーキテクチャの制約はない。

## Goals / Non-Goals

**Goals:**

- Bashコマンドの危険度をパターンマッチで5段階判定し、ステータスバーに即座に表示する
- フック登録・スクリプト配置を拡張のactivate/deactivateで自動管理する
- ユーザーの設定作業をゼロにする（インストールだけで動く）

**Non-Goals:**

- Bash以外のツール（Read, Edit, Write等）の危険度判定
- LLMベースの高度な判定
- 自動承認/自動ブロック制御
- 判定ルールのカスタマイズUI
- 危険度履歴のサイドパネル

## Decisions

### 1. フックと拡張の通信方式: ファイルベースIPC

**決定**: `/tmp/claude-danger-state.json` に判定結果を書き込み、VSCode拡張が `fs.watch` で監視する。

**代替案**:
- WebSocket / HTTP — フック側がワンショット実行のため、サーバー維持のオーバーヘッドが不釣り合い
- VSCode IPC — 拡張APIから直接フックを呼ぶ手段がない

**理由**: フックスクリプトはコマンド実行ごとに独立プロセスで起動される。ファイル書き込みは最もシンプルで追加依存がない。`$TMPDIR` ではなく `/tmp` を使うのは、macOSの `$TMPDIR` がセッションごとに異なるパスを返し、フックと拡張で一致させるのが煩雑なため。

### 2. 判定ロジックの実装言語: Node.js

**決定**: フック判定スクリプトを Node.js で実装する。

**代替案**:
- シェルスクリプト — 正規表現の記述・メンテナンスが困難
- Python — 別途インストールが必要な環境がある

**理由**: VSCode が Node.js ランタイムを同梱しており、別途インストール不要。正規表現パターンの管理もJavaScriptが自然。

### 3. 危険度判定の評価順序: 高レベルから降順

**決定**: Lv.5 → Lv.4 → Lv.3 → Lv.1 の順にパターンマッチし、最初にヒットしたレベルを採用。どれにも該当しなければ Lv.2（デフォルト）。

**理由**: 安全側に倒す設計。`rm` は Lv.3 だが `rm -rf` は Lv.5 のように、より危険なパターンを先に評価することで、包含関係のあるパターンを正しく処理できる。未知のコマンドは「副作用ありうる」としてLv.2にすることで、偽陰性（危険なのに安全と表示）を抑制する。

### 4. フック登録の管理方式: 拡張が直接 settings.json を編集

**決定**: `~/.claude/settings.json` を拡張のactivate時にJSON操作で編集し、`hooks.PreToolUse` 配列にエントリを追加する。deactivate時に削除する。

**代替案**:
- ユーザーに手動設定させる — UXが悪い
- CLIコマンドで登録 — Claude Code CLIにフック管理コマンドがない

**理由**: ゼロコンフィグを実現するため。既存のフック設定を壊さないよう、配列への追加/特定エントリの削除のみ行う。

### 5. ステータスバーの表示内容と位置

**決定**: ステータスバー左側に `🟢 Lv.1 cat README.md` 形式で表示。背景色をレベルに応じて変更。クリックで詳細ポップアップ。

**理由**: 左側は作業コンテキスト情報の定位置。コマンドの先頭部分を併記することで、何のコマンドかを即座に把握できる。

### 6. 状態ファイルのスキーマ

**決定**:
```json
{
  "level": 3,
  "command": "rm temp.log",
  "matchedPattern": "\\brm\\b",
  "timestamp": 1710000000000
}
```

**理由**: 判定に必要な最小限の情報。`timestamp` により古い状態の検知が可能。`matchedPattern` はクリック時の詳細表示で「なぜこのレベルか」を説明するために使用。

## Risks / Trade-offs

- **パターンマッチの限界** → パイプ・サブシェル・変数展開内のコマンドは完全には解析できない。MVPでは割り切り、明らかに危険なパターンの検知に集中する。将来的にはシェルパーサーの導入を検討。

- **`/tmp` の共有性** → 他プロセスが同名ファイルを操作する可能性がある。ファイル名にプレフィックス `claude-danger-` を付けて衝突リスクを軽減。セキュリティ上の懸念（改ざん）はMVPでは許容する（表示専用で制御には使わないため）。

- **settings.json の同時編集** → 拡張と他のプロセスが同時に `~/.claude/settings.json` を編集すると競合する。activate/deactivate時の一回のみの操作なので頻度は低い。read → modify → write をアトミックに近い形で行う。

- **fs.watch の信頼性** → OSによっては `fs.watch` のイベントが重複・欠落することがある。タイムスタンプベースの重複排除と、フォールバックとしてのポーリング（数秒間隔）で対応。
